Android Fundamentals

  • Activity Lifecycle
    • An activity lifecycle extends from the creation of an activity to its destruction, when the system reclaims that activity’s resources. As the user navigates in and out of an activity, each activity transitions between different states in the activity lifecycle.
    • While an Android app can have multiple activities, it is recommended to have a single activity.
    • The Activity class, along with all of its subclasses, implement a set of callback methods when the activity moves from one state to another. These callbacks can be overridden in order to customise behaviour.
    • onCreate() when the system creates the app
      • is only called once during the LifeCycle to initialise the app
    • onStart() makes the app visible on screen (no interaction)
    • onResume() brings app to the foreground (interactable)
    • onPause() causes the app to lose focus (no interaction)
      • instances include when the share button is pressed - the app is still visible but no longer interactive
    • onStop() causes the app to no longer be visible
    • onRestart() is called when the activity has already been Stopped - this is the place to put code to run when the activity is NOT being started for the FIRST time
    • onDestroy() to close all objects the activity was using and reclaim memory
      • this is also called when the app’s code calls finish() or the system needs to destroy and recreate the activity because of a configuration change
    • When you call the onCreate() method, you must call the superclass implementation to complete the Activity, so you must immediately call super.onCreate(). The same is true for all other lifecycle callbacks.
      • A good way to see the methods that can be overridden within a class it to place the cursor within the class and use shortcut ctrl+o to see the methods that can be overridden.
    • A good convention is to declare a TAG constant at the top of the MainActivity.kt file, above the class declaration.
      • private const val TAG = "MainActivity"
  • Configuration Changes:
    • While remember allows Compose to recall state during recompositions, it does not retain this state during configuration changes. For this, the function rememberSaveable is required, which saves values across recompositions AND configuration changes.
  • Logging:
    • in order to log:
      • Log.[priority](TAG, "description")
        • Where [priority]:
          • v: verbose
          • d: debug
          • i: information
          • w: warning
          • e: error
    • The Log class writes messages to the Logcat

Testing

The test directories must be a mirror of the app directory in order for the tests to function correctly. However, if you are needing to alter a file for the sake of testing, such as changing the data set to simplify testing protocols, you can sub out the original file with the data test case file in the test directory.

Note that test methods are executed in isolation to avoid unexpected side effects from mutable test instance states. By default, before each test method is executed, JUnit creates a new instance of the test class. Therefore, a class instance is NOT used multiple times across different tests but instead, a new instance of that class is created with every test. It is therefore okay to use something like: private val viewModel = AppViewModel() across multiple tests, without worrying that the subsequent tests will be affected by the state left behind after previous tests. Note that this ‘per-method test instance lifecycle is the default behaviour since JUnit 4.

Test Strategy

  • Success Path
    • AKA happy path tests, for functionality for a positive flow. A positive flow has no exception or error conditions
  • Error Path
    • Functionality for negative flow - test how the app responds to error conditions or invalid user input
  • Boundary Case
    • Testing the start and end, minimum and maximum boundaries of the use cases of the app.
  • Tests should be:
    • Focussed
      • Test a unit, such as a piece of code. This should be narrow with the focus on validating the correctness of individual pieces of code, not the app as a whole or multiple pieces of code at the same time.
    • Understandable
      • At a glance a developer should be able to immediately understand the intention behind the test
    • Deterministic
      • The test should consistently pass or fail, when run multiple times, without code changes.
    • Self-contained
      • Should not require any human interaction or setup and should run in isolation

Code Coverage

Code coverage plays a vital role to determine if you adequately test the classes, methods, and lines of code that make up your app. Android Studio provides a test coverage tool for local unit tests to track the percentage and areas of your app code that your unit tests covered. Note that code coverage indicates the percentage of code executed by your tests but it does not indicate that the code is verified. If all assert commands were removed from a test kit, so long as the lines remained that run the code, the coverage could still be 100%. It is therefore important to consider code coverage as a factor in your test coverage, but not the single marker of effectiveness. A low code coverage is often best used as an indicator to find parts of code that have been neglected by your tests.
